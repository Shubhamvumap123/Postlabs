## 2024-05-18 - Custom Dialog Accessibility
**Learning:** Custom modal components built with portals often lack critical accessibility attributes (like `role="dialog"`, `aria-modal="true"`) and keyboard bindings (like closing on `Escape` key), making them completely inaccessible to screen reader users and those navigating via keyboard. Additionally, providing ARIA associations (`aria-labelledby` and `aria-describedby`) for titles and descriptions is crucial for providing context.
**Action:** When building or auditing custom portal-based modals in the future, I must always ensure they implement proper ARIA roles (`role="dialog"`, `aria-modal="true"`), map titles and descriptions using React's `useId()` for `aria-labelledby` and `aria-describedby`, and include a global `document` listener to ensure they close on the `Escape` key. Icon-only close buttons should also have an explicit `aria-label` and `focus-visible` styling.
## 2025-03-20 - Newsletter Form Accessibility Improvements
**Learning:** Using `placeholder` text as the only label for input fields is an accessibility anti-pattern. While visual proximity implies the field's purpose, screen readers and assistive technologies require explicit `<label>` associations. Additionally, icon-only or symbol-based buttons (like "→" for submit) lack context without an `aria-label`.
**Action:** When creating forms, always ensure every `<input>` has an associated `<label>`, using the `.sr-only` class if the label needs to be visually hidden but available to screen readers. For symbol-based buttons, provide a clear `aria-label` describing the action, and always implement explicit `:focus-visible` styles for keyboard navigation.
## 2025-05-18 - Custom Toggle Switch Accessibility
**Learning:** Building custom toggle switches using `<div>` elements with `onClick` handlers removes keyboard accessibility and screen reader support. Users navigating via keyboard cannot focus or activate the toggle, and screen readers do not recognize its state or purpose.
**Action:** When creating custom interactive elements like toggles, always use a native interactive element (e.g., `<button type="button">`) with the appropriate ARIA roles (e.g., `role="switch"`), state attributes (e.g., `aria-checked`), and associated labels (`aria-labelledby` or `aria-label`). Ensure the element has clear `:focus-visible` styles for keyboard navigation.
## 2025-05-18 - Form Autocomplete and Loading State UX
**Learning:** Standard text inputs in custom React components do not inherently help users fill them out. Adding `autoComplete` attributes dramatically speeds up form completion and meets key accessibility guidelines (WCAG 1.3.5 Identify Input Purpose). Additionally, simple text updates like "Sending..." on buttons are less noticeable than standard visual patterns like spinning loaders, which give immediate, unambiguous feedback that an async action is processing.
**Action:** Always include standard `autoComplete` attributes (e.g., `given-name`, `email`) on common form fields, and always use an animated spinner icon alongside text for interactive loading states to provide immediate visual feedback.
